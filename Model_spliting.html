Why Model Splitting and Parallelism in LLMs?

🚫 Problem

Large Language Models (LLMs) like GPT-3 or GPT-4 have hundreds of billions of parameters (hundreds of GBs), making it impossible to fit on a single GPU.

🎯 Goal

Efficiently train and infer with LLMs by splitting computation and memory load across multiple devices.

⚡ Challenges

  • Model size vs GPU memory
  • Communication overhead
  • Load balancing
  • Minimizing idle time
✅

High-Level Categorization

Model Splitting and Parallelism can be grouped into four main strategies:

A. Data Parallelism

What: Duplicate entire model on multiple GPUs; each GPU processes different mini-batch.

Pros: Simple (PyTorch DistributedDataParallel).

Cons: Requires full model copy → large memory.

Example: GPT-2 on 4 GPUs, each processes 16 token sequences.

C. Expert Parallelism

What: Multiple experts; each token goes to subset of experts.

Example: Switch Transformer.

Benefit: Activates only part of model → efficiency.

Challenge: Load balancing between experts.

B. Model Parallelism

Split model parameters/layers across GPUs:

1. Tensor (Intra-Layer)

Split within single layer. Each GPU holds part of attention weights. Used in Megatron-LM.

Trade-off: No idle time ↔ High communication.

2. Pipeline Parallelism

Split layers into stages: GPU1→Layers 0-11, GPU2→12-23, etc.

Challenge: Pipeline bubble. Fix: Micro-batching.

D. Sequence Parallelism

What: Split input sequence tokens across GPUs to reduce activation memory.

Example: 2048 tokens on 4 GPUs = 512 tokens each.

Used in: Megatron-LM with ZeRO optimizations.

Part 2: Single Device Acceleration (1/2)

A. Algorithmic Optimizations

1. Low-Rank Approximation

Replace large weight matrices with low-rank factors.

Example: LoRA for fine-tuning.

2. Quantization

Reduce precision: FP32 → FP16 → INT8 → 4-bit.

Examples: GPTQ, BitsAndBytes

Impact: Up to 4x memory reduction

3. Pruning

Remove unimportant weights or attention heads.

Example: SparseGPT

4. Knowledge Distillation

Train smaller student from large teacher model.

Examples: DistilBERT, TinyLlama

B. Compute & Kernel Optimizations

1. Fused Kernels

Merge multiple CUDA kernels to reduce launch overhead.

Example: FlashAttention (Tri Dao et al.)

Impact: 2-4x speedup in attention

2. Better Memory Layout

Use tensor cores effectively (FP16/FP8).

Align data for GPU-friendly access patterns.

Example: NVIDIA cuBLAS with mixed precision

3. Operator Reordering

Move computations to reduce memory bandwidth.

Example: Pre-compute LayerNorm stats

✅

Part 2: Single Device Acceleration (2/2)

C. Memory Management

1. Activation Checkpointing

Save memory by recomputing activations during backprop.

Example: PyTorch torch.utils.checkpoint

2. Paged Attention

Optimize KV cache using paged memory for long sequences.

Example: vLLM framework (Zheng et al., 2023)

Impact: Large throughput improvement

3. Memory Offloading

Move tensors between GPU and CPU memory (or SSD).

Example: HuggingFace Accelerate with offload

D. Inference Acceleration

1. Speculative Decoding

Draft model predicts tokens → main model verifies.

Example: SpecInfer (Chen et al., 2023)

Impact: 2-3x faster inference

2. KV Cache Optimizations

Store key-value pairs efficiently across attention layers.

Examples: PagedAttention in vLLM, Quantized KV cache

3. Dynamic Batching

Combine requests in real time to improve GPU utilization.

Examples: vLLM, TensorRT-LLM

Processing Methods Evaluated

1️⃣ Sequential Processing

Strategy: Single model instance utilizing all available CPU cores with optimized threading.

llm = Llama(
  n_threads=8,  # All CPU cores
  n_ctx=2048,   # Full context window
  numa=True,    # NUMA optimization
  use_mmap=True, # Memory mapping
  f16_kv=True   # Efficient cache
)

Advantages: Maximum resource utilization, no overhead, layer splitting optimizations

2️⃣ Process Parallel

Strategy: Multiple separate processes, each loading independent model instances.

# 2 workers, each with limited resources
ProcessPoolExecutor(max_workers=2)
# Per process: n_threads=2, n_ctx=1024

Theory: Parallel processing should leverage multiple CPU cores simultaneously

Reality: Resource contention and memory overhead

3️⃣ Thread Parallel

Strategy: Single shared model with thread-safe access using locks.

# Shared model with thread synchronization
with self.lock:
  # Thread-safe access
  response = self.llm(prompt,
                     max_tokens=50)

Trade-off: Shared resources vs serialization overhead

4️⃣ Batch Processing

Strategy: Optimized sequential processing with batching optimizations.

llm = Llama(
  n_batch=256,   # Optimized batch size
  n_ubatch=128  # Micro-batch optimization
)

Goal: Improved throughput through batch optimizations

🎯 Hypothesis: Each method tests different parallelization strategies to identify optimal CPU utilization patterns for LLM inference on multi-core systems.
